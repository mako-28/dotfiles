# 作業計画・ログの運用ルール

---

## 目的

作業の計画と実施記録を `.log/` ディレクトリに残し、以下を実現する:

- 作業の再現性と追跡可能性を確保する
- 方針決定の経緯を後から参照できるようにする
- 複数セッションにまたがる作業の引き継ぎを容易にする

---

## 出力先

```
.log/[sub-directory]/
```

- `sub-directory` はユーザーが指定する
- **ユーザーから指定されていない場合は、自分で決めずに必ず質問すること**
- **命名は `YYYYMMDD-{名前}` 形式を推奨**（例: `20260629-user-login`）。着手日をプレフィックスに
  付けることで、`.log/` 直下を時系列順（辞書順＝時系列順）に並べられる。日付は「その作業群を
  始めた日」で固定する。

### Git 管理対象外

`.log/` は `.gitignore` で除外されている。**plan / work / session の各ファイルは commit 対象に含めないこと**。
`git add` 時に `.log/` 配下を誤ってステージしない（個別パス指定で add するなど）。

---

## ファイル構成

1つの作業につき、以下の2ファイルを作成する:

| ファイル | 命名規則 | 内容 |
|---------|---------|------|
| 計画 | `{番号}-{作業名}-plan.md` | 背景・方針・設計・リスク |
| 作業ログ | `{番号}-{作業名}-work.md` | 実施内容・結果・知見 |

- `{番号}` は同一ディレクトリ内で連番（既存ファイルの最大番号 + 1）
- `{作業名}` はケバブケースで簡潔に（例: `user-login-validation`）

中間ファイル・一時ファイルはサブディレクトリ直下に置かず、作業ごとに `{番号}-tmp/` にまとめる
（`{番号}` を plan / work と揃えることで、どの作業の中間ファイルかが一目で分かる）:

```
.log/20260629-user-login/
  ├── 1-validation-plan.md
  ├── 1-validation-work.md
  ├── 1-tmp/            ← 作業 1 の中間ファイル・一時ファイル
  │   ├── extracted.tsv
  │   └── hashes.txt
  ├── 2-session-handling-plan.md
  ├── 2-session-handling-work.md
  └── 2-tmp/            ← 作業 2 の中間ファイル・一時ファイル
```

---

## plan ファイルの構成

```markdown
# 実行計画: {作業タイトル}

作成日: YYYY-MM-DD
対象ブランチ: `{ブランチ名}`
参照: {関連ドキュメント、コミット}

---

## 背景
（なぜこの作業が必要か）

## 検討した方式と却下理由
（採用しなかった案とその理由）

## 採用方式
（選んだ方式とその利点）

## 設計
（ファイル構成、コード例、インターフェース）

## 型安全性
（型レベルでの保証内容）

## リスクと対策
（想定リスクとその対処法）

## 技術的な注意事項
（実装時に気をつけるべきこと）
```

全てのセクションが必須ではない。作業の規模に応じて取捨選択する。

---

## work ファイルの構成

```markdown
# 作業ログ: {作業タイトル}

作業日: YYYY-MM-DD
対象ブランチ: `{ブランチ名}`
実施計画: `{対応するplanファイル名}`
対象コミット: `{関連コミット}`

---

## 実施内容

### 作成・変更ファイル一覧
（★新規 / ★変更 の区別付き）

### 各ファイルの変更概要
（何をしたかを簡潔に）

---

## 検証結果
（型チェック、ビルド、動作確認の結果）

---

## 技術的発見・知見
（作業中に判明した技術的事項。将来の参考になるもの）
```

## 運用ルール

### 作成タイミング

- **plan**: 作業開始前に作成する。方針検討・設計フェーズの成果物として。
- **work**: 作業完了後に作成する。実施内容の記録として。

### 作業途中の状態

- plan に「作業順序」セクションを含め、進捗を `✅ 完了` / `(未着手)` で管理してよい
- ただし作業が完了したら、進捗管理は plan から削除し、work に最終結果のみを記載する

### plan と work の分離基準

| plan に書くこと | work に書くこと |
|---------------|---------------|
| なぜその方式を選んだか | 実際に何をしたか |
| 検討した代替案 | 作成・変更したファイル一覧 |
| 設計（ファイル構成、コード例） | 検証結果（型チェック、ビルド） |
| リスクと対策 | 技術的発見・知見 |
| 技術的注意事項 | 想定外の問題とその対処 |

### 既存のセッションログ

過去に `*-session.md` として作成されたファイルは、会話の生ログとして残してよい。
今後の新規作業では plan / work の2ファイル構成を使う。

### 中間ファイルを削除しない（保存の原則）

`.log/` 配下に作成したファイルは、作業の記録として**原則すべて残す**。勝手に削除しないこと。

- plan / work の md だけでなく、調査・分析の過程で生成した**中間ファイル**（`*.tsv`、`*.txt`、抽出結果、スクリプト出力、ハッシュ一覧など）も対象に含む。
- 中間ファイル・作業用の一時ファイルは、サブディレクトリ直下に散らかさず **作業ごとに `{番号}-tmp/` 配下にまとめて出力する**（`{番号}` を plan / work と揃え、md とは分離する）。
- これらは「どんな作業・判断を行ったか」の根拠であり、消すとその経緯が失われる。
- 整理したい場合でも自分の判断で消さず、**残すことを既定**とし、削除が必要ならユーザーに確認する。
- `.log/` は git 管理対象外（前述）なので、残してもリポジトリは汚れない。
