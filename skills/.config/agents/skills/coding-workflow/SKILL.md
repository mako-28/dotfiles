# 作業の進め方ルール

---

## 原則

ユーザーからの作業依頼を受けたら、**実装に着手する前に必ず計画を立て、合意を得る**こと。

---

## 手順

### 1. 調査・分析

- 対象コードを読み、現状を把握する
- 影響範囲を特定する
- 不明点があればユーザーに質問する

### 2. 計画の作成

- `.log/[sub-directory]/` に plan ファイルを作成する
  - sub-directory が未指定の場合は**ユーザーに質問する**
- 計画には以下を含める:
  - 背景（何が問題か）
  - 検討した方式と選定理由
  - 設計（ファイル構成、変更概要）
  - リスク
- plan / work ファイルの命名規則・記載項目の詳細は `work-log` スキルを参照する。

### 3. ユーザーとの合意

- plan ファイルの内容をユーザーに提示する
- ユーザーが承認するまで実装に着手しない
- ユーザーから修正指示があれば plan を更新して再提示する

### 4. 実装

- 合意した計画に基づいて実装を進める
- 計画から大きく逸脱する場合は、一旦止まってユーザーに相談する

### 5. 記録

- 作業完了後に work ファイルを作成する
- 詳細は `work-log` スキルを参照

---

## 例外

以下の場合は plan ファイルなしで実装に着手してよい:

- 明らかに単純な変更（1ファイル数行の修正、typo修正等）
- ユーザーが「計画不要」「すぐやって」等と明示的に指示した場合
- 調査・検証のみの依頼（結果を報告するだけで実装を伴わない）

---

## やってはいけないこと

- 計画の合意なしに大規模な実装を始めること
- ユーザーの承認前に複数ファイルにまたがる変更をコミットすること
- 計画と異なる方式で実装して事後報告すること
