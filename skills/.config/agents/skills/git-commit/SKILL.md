# Git コミットルール

## 原則
1コミット = 1つの完結した論理的変更（1つの目的）にする。レビュー・revert・履歴追跡が
しやすい単位を意識する。

## メッセージの書き方
- 基本は Conventional Commits 形式に従う: `type(scope): 説明`
- `scope` には対象モジュール名を入れる。リポジトリ全体・横断的な変更など特定モジュールに
  紐づかない場合は省略可。
- 説明（subject）は簡潔に「何をしたか」が分かるように書く。末尾に句点（。）は付けない。
- 必要に応じて本文（body）を付け、「なぜ」その変更をしたか・補足を書く。
  - subject と body は空行で区切る。
- **既存の慣習を最優先する**。コミット前に `git log --oneline` を確認し、
  プレフィックスや言語（日本語/英語）の慣習を踏襲する。
- よく使う type:
  - `feat`: 新機能
  - `fix`: バグ修正
  - `refactor`: 挙動を変えないコード改善
  - `test`: テストの追加・修正
  - `build`: ビルド設定・依存関係の変更
  - `docs`: ドキュメント・ルールの変更
  - `chore`: その他の雑務

## 粒度は細かめにする

| やること | 説明 |
|---------|------|
| 目的ごとに分ける | 「テスト追加」と「ドキュメント化」など、別系統の変更は別コミットにする |
| 実装とリファクタを混ぜない | 機能追加と無関係な整形・リネームは分離する |
| フォーマットとロジックを分ける | 「フォーマットだけの変更」と「ロジックの変更」は別コミットにする |
| 関連ファイルはまとめる | 同一目的の複数ファイル（実装＋付随ファイル等）は1コミットでよい |

### やってはいけないこと
- 複数の無関係な変更を1コミットに詰め込む（"fix several things" 等）
- 1つの論理変更を意味なく細切れにする（粒度を細かくする＝過剰分割ではない）

### 抽出リファクタリングは feat → refactor の2コミットに分ける
新しいクラス（parser / mapper / domain record など）を抽出して既存コードを委譲へ縮小するときは、
次の2コミットに分ける。

1. `feat(scope): add {新クラス} with tests` — 新クラスとそのテストを追加（既存コードはまだ変更しない）。
2. `refactor(scope): delegate {既存クラス} ... to {新クラス}` — 既存クラスを新クラスへの委譲に縮小する。

各コミット時点でビルド・テストがグリーンであること（壊れた中間状態を作らない）。
段階が多い場合（parser 抽出 → mapper 抽出 など）は、段ごとに feat/refactor のペアを繰り返す。

## ライブラリ（依存関係）の扱い
- **ライブラリの追加**は、それ単体でコミットする。
  - 依存追加のみをまず1コミットにし、その後で利用コードを別コミットにする。
- **ライブラリの更新（バージョン変更）**は、それに伴う影響コードの変更を含めてコミットする。
  - 更新によって必要になった API 追従・修正は同じコミットにまとめ、ビルドが通る状態を保つ。
- ライブラリの削除は、その依存を使わなくする変更が完了した後に行う。

## ステージング
- コミット前に `git status` と `git diff` で内容を必ず確認する。
- `git add -A` や `git add .` で無差別にステージしない。**変更内容を確認し、目的ごとに
  個別パスで `git add` する**。
- 意図したファイルだけをステージする。無関係なファイル（ログ・生成物・一時ファイル等）を含めない。
- **`.log/` 配下は commit 対象外**（`.gitignore` 済み）。plan / work / session を
  誤ってステージしないこと。詳細は `work-log` スキルを参照。

## コミットのタイミング
- commit / amend / push / PR 作成は **ユーザーから明示的に依頼されたときのみ** 行う。

## 一般的なルール
- 機密情報（パスワード・トークン・鍵など）を絶対にコミットしない。
- 各コミットはビルド・テストが通る状態にする（壊れた中間状態をコミットしない）。
- コミットが失敗した・フック（hook）に弾かれた場合は、修正して新しいコミットを作る。
  失敗したコミットを `--amend` で取り繕わない。
- 履歴の書き換え（force push、rebase での改変）は、共有ブランチでは行わない。
- 空コミットや、フックのスキップ（`--no-verify`）は明示的な指示がない限り行わない。
